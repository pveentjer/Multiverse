Multiverse 0.7

Multiverse 0.7 was released at 1 May 2011 and was a total rewrite from the 0.6 release. The main features are:
  1. Removal of the central clock. The STM design before 0.7 was based on MVCC (TL2 to be more specific) and although it is really amazing, and readonly transactions can be made really cheap, update transaction still need to contend for the central clock to increment. There are some optimizations that can be applied to remove contention on this clock, but the clock is still required. The STM of Multiverse 0.7 is based around the idea of the SkySTM where no central clock is needed.
  2. Isolation Levels. Multiverse 0.7 supports the following isolation levels:
    • read committed: where a transaction will only see committed state, but no other consistency guarantees are provided. Read committed isolation level allows for much higher concurrency (it won't fail unlike the other isolation levels) and therefor scales better, at the cost of reduced consistency. The dirty read isolation level is not supported since a transaction will never see changes made by another transaction.
    • repeatable read:
    • snapshot: the Oracle version of the Serialized isolation level
    • serialized: provides a truly serialized execution of transaction where the writeskew problem is not allowed. explicit support for enabling/disabling the writeskew problem from happening has been removed
  3. Pessimistic locking: to compensate for the optimistic behavior STM's normally provide. It can be done on the reference level using ensure/privatize, but can also be done on the transaction level using the following pessimistic lock levels:
    • None: default. Nothing is locked during the execution of the transaction.
    • Ensure writes: when this pessimistic lock level is used, you essentially get the same behavior as in the Oracle database, where each update/delete/insert automatically is locked for the remaining duration of the transaction.
    • Ensure reads
    • Privatize writes
    • Privatize reads
    Other added features are accessing the current locking status of a reference (to see if it ensured/privatized by itself or by others).
  4. Deferred ensure: makes it possible to deal optimistically with the writeskew problem on the ref level without causing overhead on all ref/transactional objects.
  5. Improved performance: a lot of effort was spend on improving the performance of the STM. One of the big improvements was the removal of all unwanted object creation by adding an object pool to each transaction (which also can be reused).
  6. Atomic operations on refs the api of the references has been improved greatly and one of the new features is support for atomic operations. Atomic operations are easier to do and a lot faster and can be a better solution than using a transaction where only one ref is updated. The following atomic operations are available:
    • atomicGet
    • atomicWeakGet
    • atomicGetAndSet
    • atomicSetAndGet
    • atomicAlterAndGet
    • atomicGetAndAlter
    • atomicCompareAndSet
    • atomicIsNull
    • atomicIsFree
    • atomicIsPrivatized
    • atomicIsEnsured
    • atomicIncrementAndGet
    • atomicGetAndIncrement
  7. Alter
  8. Commute:
  9. Transaction.abortOnly

Removal of functionality

Some features have been removed from the 0.7 release:
  1. Instrumentation: will be added in the future again
  2. TransactionalExecutor: will be added in the future
  3. Transactional datastructures: will be added in the 0.8 release
  4. Groovy support: we are working on integrating Multiverse directly with Groovy/GPars
  5. Scala support: all Scala support is done through the Akka project and we are planning to provide an implementation that implements the Scala STM API
